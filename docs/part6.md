# Part 6：副作用とEffect

Part 2 で我々は、例外や状態を「型に現れていないもの」として見た。

Part 4 では、それらを文脈つきの計算として合成するために Monad まで進んだ。

Part 2 でいう effect は、文脈つき計算一般を指していた。
この Part ではその中でも、外部世界と関わる effect に焦点を当てる。

では次に問うべきは何か。

> 副作用とは、結局何なのか

この Part では、副作用を道徳的に嫌うのではなく、構造として読む。

重要なのは「副作用をなくすこと」ではない。

> 副作用を、どこで・いつ・どの形で実行するかを設計すること

である。

---

## Chapter 36：副作用とは何か

### 問題提起

副作用という言葉は、しばしば曖昧に使われる。

- 画面表示
- ログ出力
- HTTP通信
- DB更新
- 時刻取得

これらは全部「副作用」と呼ばれる。

だが、ただ副作用と呼ぶだけでは設計が進まない。

必要なのは、その本質をつかむことだ。

### 直感的説明

副作用とは「値を返すこと以外の何か」ではない。

より正確には、

- 外部世界に依存し
- 外部世界を変化させうる
- 同じ式を置き換えても同じ意味にならない

計算である。

純粋関数なら、`f(x)` をその結果で置き換えても意味は変わらない。

しかし `Date.now()` や `fetch()` はそうではない。

つまり副作用の問題は、実行することそのものより、置換可能性が壊れることにある。

### 抽象の導入

ここで重要なのは、実行と記述を分けることである。

「今ここで実行する」のではなく、

> 実行されうる計算を、まず値として表現する

という方向へ進む。

この瞬間、副作用は制御不能な混入物ではなく、Effect という構造になる。

### TypeScriptコード

```ts
const now = (): number =>
  Date.now()

const print = (message: string): void => {
  console.log(message)
}

const save = async (body: string): Promise<void> => {
  await fetch("/api/save", {
    method: "POST",
    body,
  })
}
```

これらは「何かを返す関数」でもあるが、それ以上に外部世界への窓である。

だからこそ、通常の `A -> B` と同じ顔をさせたままでは危うい。

### 数学的補足

純粋性の重要さは、道徳ではなく等式推論にある。

置換可能性が保たれる世界では、式変形がそのまま設計の道具になる。

副作用はこの性質を壊しうるため、構造化が必要になる。

### まとめ

副作用とは「汚いコード」ではない。

外部世界との接続であり、だからこそ境界として扱わなければならない。

## Chapter 37：遅延評価

### 問題提起

副作用を扱いにくくする最大の理由は、書いた瞬間に実行されることではない。

むしろ、「いつ実行されるか」が構造に出ていないことだ。

たとえば次のコードを考える。

```ts
const userPromise = fetch("/api/user")
```

この一行を書いた時点で、ネットワーク通信は始まる。

だが設計上は、まだ「欲しい計算を宣言しただけ」であってほしいことが多い。

### 直感的説明

遅延とは、怠けることではない。

実行のタイミングを値の生成から切り離すことである。

すぐに実行するのではなく、「あとで実行できる計算」として持つ。

この分離により、

- 合成してから実行する
- リトライ方針をつけてから実行する
- ログやトレースを挿してから実行する

ことが可能になる。

### 抽象の導入

遅延評価を導入すると、計算は

```text
値を返すもの
```

ではなく

```text
実行すれば値を返す記述
```

になる。

この違いは大きい。

副作用を「その場で起きる出来事」から、「後で解釈されるプログラム」へ変えるからだ。

### TypeScriptコード

```ts
type Lazy<A> = () => A

const now: Lazy<number> = () =>
  Date.now()

const greeting: Lazy<string> = () =>
  `generated at ${now()}`
```

この例は単純だが、遅延の本質は出ている。

`number` を持っているのではない。

`number` を得る手続きを、まだ実行せずに持っている。

### 数学的補足

遅延はしばしば thunk として表現される。

重要なのは、計算の意味と計算の実行を分けることだ。

その分離によって、副作用は合成前に観察・変換・注釈づけできる対象になる。

### まとめ

副作用を制御したいなら、まず即時実行から距離を取る必要がある。

遅延は、そのための最初の手段である。

## Chapter 38：Effectシステム

### 問題提起

`Promise` だけでは足りない場面がある。

なぜなら `Promise` は非同期性は表せても、

- 失敗の型
- 必要な依存
- キャンセル
- リトライ
- ログやトレース

を構造としては十分に表さないからである。

現実のアプリケーションでは、必要なのは「非同期」であること以上に「どんな文脈で実行されるか」である。

### 直感的説明

Effect システムは、副作用を隠すための技術ではない。

副作用に付随する条件を、型へ押し戻す技術である。

つまり、

- 何が失敗しうるか
- 何に依存しているか
- 何を実行しようとしているか

を、単なるコメントや規約ではなく、接続面そのものへ埋め戻す。

### 抽象の導入

Effect を抽象化すると、計算はしばしば次のような形で読める。

```text
Effect<依存, 失敗, 成功>
```

この形が強いのは、成功値だけでなく、

- 必要条件
- 失敗条件
- 実行文脈

まで合成可能な対象として扱えるからである。

### TypeScriptコード

```ts
type Effect<R, E, A> = {
  readonly _R: (_: R) => void
  readonly _E: () => E
  readonly _A: () => A
}

type Config = {
  apiBaseUrl: string
}

type HttpError = {
  message: string
}

type User = {
  id: string
  name: string
}

type LoadUser = Effect<Config, HttpError, User>
```

これは実装ではなく形の説明である。

ここでは概念図として `Effect<R, E, A>` と書き、
`依存・失敗・成功` の順で読んでいる。

ただし `effect` ライブラリの実際の API では、
型引数は `Effect<A, E, R>` の順で表される。

本書では構造を理解しやすくするために前者の順で説明し、
実コードとの接続時に後者へ読み替える。

重要なのは、`User` を返すことしか見えなかった計算が、依存と失敗を伴う構造として読めるようになることだ。

### 数学的補足

Effect システムは、Functor や Monad の拡張的応用として理解できる。

ただし本質はライブラリ名ではない。

計算の文脈を成功値の外へ逃がさず、型で合成可能にすることである。

### まとめ

Effect システムは豪華な非同期ライブラリではない。

隠れていた条件を型へ戻すための装置である。

## Chapter 39：effect-tsとの接続

### 問題提起

ここまでの話を、実際の TypeScript 生態系へ接続する。

抽象だけでは実務にならない。

だが実務に降りるときも、思想を捨てる必要はない。

`effect-ts` は、その接続点として非常にわかりやすい。

### 直感的説明

`effect-ts` が提供しているのは、便利関数の寄せ集めではない。

「副作用を値として持ち、合成してから実行する」という設計原理である。

この原理に乗ると、

- 依存注入
- 失敗の型付け
- リソース管理
- 並行処理

が、一つの計算モデルの上で整理される。

### 抽象の導入

本書の文脈で `effect-ts` を読むなら、次のようになる。

- `Option` は欠落の構造
- `Either` / `Result` は失敗の構造
- `Effect` は副作用を含む計算の構造

つまり、これまでの議論を実務レベルの粒度へ引き上げたものとして読める。

### TypeScriptコード

```ts
import { Effect } from "effect"

type Config = {
  apiBaseUrl: string
}

type User = {
  id: string
  name: string
}

declare const loadUser: (
  id: string
) => Effect.Effect<User, Error, Config>

declare const renderUser: (user: User) => string

const program = Effect.map(loadUser("user-1"), renderUser)
```

ここで起きていることは、これまでと同じである。

裸の値ではなく文脈つきの値に対して、写像を適用している。

必要になれば `flatMap` で次の Effect へ接続する。

### 数学的補足

ライブラリは実装詳細を持つ。

しかし、学ぶべきなのは API の暗記ではない。

どの API も結局は、Functor・Monad・環境・失敗・実行境界という構造的要請の具体化だと読むことである。

### まとめ

`effect-ts` は流行の道具として学ぶものではない。

副作用を構造化するとはどういうことかを、TypeScript の現場で引き受けた形として読むべきである。
