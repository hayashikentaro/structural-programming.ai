# Part 8：AI時代の開発

本書は最初から、AI 時代を前提にしていた。

なぜなら、今や「コードを書くこと」だけでは差にならないからである。

だがこの事実を、単に悲観として受け取る必要はない。

むしろ逆だ。

AI が実装を大量に生成できるからこそ、人間の仕事はより鮮明になる。

> 何を合成させるか
> どこで失敗を露出させるか
> どの境界を型で固定するか

この Part では、AI 協働開発を「便利な補助ツールの使い方」としてではなく、

> 構造責任の再配置

として捉える。

---

## Chapter 46：vibeコーディング

### 問題提起

AI と対話しながらコードを増やしていく開発は、しばしば `vibe coding` と呼ばれる。

このスタイルは速い。

だが速さは、そのまま危険でもある。

なぜなら AI は、局所的にもっともらしい接続を量産できるからだ。

すると開発者は「なんとなく動く」ものを次々につなぎ、全体構造の破綻を後から受け取ることになる。

### 直感的説明

vibe コーディングの本質的なリスクは、AI が間違うことではない。

人間が構造の責任を手放すことにある。

AI は、

- メソッドを追加する
- 型を埋める
- エラー処理をもっともらしく補う

ことができる。

しかしそれは「局所的な自然さ」であって、「全体の整合性」ではない。

したがって vibe コーディングが成立する条件は一つしかない。

> 生成前に、構造の骨組みが人間側で定まっていること

### 抽象の導入

ここで役割を分ける。

AI は局所写像の生成器である。

人間は写像圏の設計者である。

つまり、

- 対象をどう切るか
- 射をどこに置くか
- 失敗や副作用をどう型へ露出させるか

は先に人間が決める。

その上で AI に実装を流させるなら、速度は利益になる。

逆にこれが曖昧なままなら、速度は単なる破壊力になる。

### TypeScriptコード

```ts
type Input = {
  email: string
}

type ValidationError = {
  message: string
}

type ValidEmail = string & { readonly _tag: "ValidEmail" }

type Result<E, A> =
  | { type: "ok"; value: A }
  | { type: "error"; error: E }

declare const validateEmail: (
  input: Input
) => Result<ValidationError, ValidEmail>
```

この例で AI に任せるべきなのは `validateEmail` の局所実装であって、
`Input` と `ValidEmail` を分けるかどうかではない。

その分割こそが構造責任だからである。

### 数学的補足

圏論の語彙で言えば、AI は射の具体化を助ける。

しかし対象の切り方と合成規則の設計は、依然として上位の判断である。

ここを混同すると、局所最適が全体破綻へ直結する。

### まとめ

vibe コーディングは悪ではない。

だが、構造のない vibe はただの漂流である。

速く書く前に、何を固定するかを決めなければならない。

## Chapter 47：役割分担

### 問題提起

AI 協働開発が失敗するプロジェクトでは、たいてい役割分担が曖昧である。

人間が全部レビューするつもりで全部投げる。

あるいは、AI を信用しないつもりで結局全部自分で書き直す。

この往復は、もっとも消耗が大きい。

### 直感的説明

役割分担は能力差の話ではない。

責任境界の話である。

AI に向いているのは、

- 局所実装
- 定型変換
- ボイラープレートの展開
- 既存構造に沿った補完

である。

人間に残るのは、

- ドメイン境界の決定
- 型の強度の決定
- 抽象導入の妥当性判断
- レビュー基準の決定

である。

### 抽象の導入

設計上は、次のように分けるとよい。

1. 人間が構造を定義する
2. AI がその構造に沿って局所実装する
3. 人間が合成の観点でレビューする

この順序が崩れると、AI が構造を仮決めし、人間が後から追認する形になる。

それは設計ではなく、事後整理である。

### TypeScriptコード

```ts
type UserId = string & { readonly _tag: "UserId" }

type User = {
  id: UserId
  name: string
}

type LoadUser = (id: UserId) => Promise<User>

type RenderUser = (user: User) => string
```

このレベルの境界までは人間が決めるべきである。

その上で `LoadUser` の詳細実装や、`RenderUser` の補助関数展開を AI に任せる方が安定する。

### 数学的補足

設計責任とは、どの射を許し、どの射を禁止するかを決める責任である。

その責任を AI に委譲しきるなら、すでに設計の主導権を失っている。

### まとめ

よい AI 協働開発は、AI を便利に使うことではなく、責任を適切に分けることから始まる。

## Chapter 48：プロンプト設計

### 問題提起

AI に対して「いい感じにやって」と頼むと、たいてい本当に「いい感じ」なものが返る。

だが問題は、その「いい感じ」が局所的であることだ。

コードは通る。

説明も自然である。

しかし境界条件、失敗条件、状態遷移の制約は、しばしば抜け落ちる。

### 直感的説明

よいプロンプトは、自然言語の巧拙ではない。

AI に何を固定条件として守らせるかの設計である。

つまりプロンプトとは、

- 実装依頼

ではなく

- 構造制約の伝達

であるべきだ。

### 抽象の導入

この本の文脈で、よいプロンプトには最低限次が入る。

- 何が入力型か
- 何が出力型か
- 何を失敗として表すか
- 何を変更してはいけないか
- どの境界を維持したいか

これは単なる指示の丁寧さではない。

AI に「許された射の集合」を伝える作業である。

### TypeScriptコード

```ts
type PromptSpec = {
  inputBoundary: string
  outputBoundary: string
  failureMode: string
  forbiddenChanges: string[]
}

const request: PromptSpec = {
  inputBoundary: "RegistrationForm -> Result<ValidationError, RegistrationRequest>",
  outputBoundary: "Result<ValidationError, RegistrationRequest>",
  failureMode: "例外を使わず Result で表す",
  forbiddenChanges: ["src/domain 以下の型定義を弱めない"],
}
```

これはもちろん AI API の実フォーマットではない。

ただし、何を伝えるべきかはよく出ている。

プロンプトは曖昧な意図表明より、構造制約の明示に寄せた方が強い。

### 数学的補足

プロンプトは形式体系ではない。

しかし、形式化されていない要求をそのまま投げるほど、AI は統計的な自然さへ流れる。

したがって、構造を保ちたいなら、自然言語の中にできるだけ型的制約を埋め込む必要がある。

### まとめ

プロンプト設計とは、AI にうまくお願いする技術ではない。

構造を崩さずに生成させるための仕様設計である。

## Chapter 49：レビュー戦略

### 問題提起

AI 協働開発では、レビューの意味も変わる。

生成速度が上がるほど、レビューが「全部読むこと」では成立しなくなるからだ。

ここで危険なのは、見た目の自然さに負けることだ。

AI のコードは、しばしば非常に読みやすい。

だが読みやすいことと、構造的に正しいことは別である。

### 直感的説明

レビューで最初に見るべきは、スタイルではない。

次の 4 点である。

- 型境界が弱くなっていないか
- 失敗が型の外へ逃げていないか
- 状態や依存が隠れていないか
- 抽象が law を保ちそうか

つまりレビューとは、局所実装の美しさを見ることではない。

合成条件が壊れていないかを見ることだ。

### 抽象の導入

AI 時代のレビューは、行単位の添削から、構造監査へ寄る。

具体的には、

1. 入出力型を見る
2. 失敗表現を見る
3. effect 境界を見る
4. law や不変条件の破れを見る

という順で見る方がよい。

これは本書全体の復習でもある。

レビューとは、結局

> その変更は、合成可能性を強めたか、弱めたか

を問う作業だからだ。

### TypeScriptコード

```ts
type ReviewChecklist = {
  inputOutputBoundary: boolean
  failuresAreTyped: boolean
  hiddenStateExists: boolean
  invariantsPreserved: boolean
}
```

このような素朴なチェックリストでも、レビューの視線はかなり変わる。

コメントの細部より先に、構造の破綻を拾えるようになるからだ。

### 数学的補足

レビューを圏論で説明しきる必要はない。

しかし「射の合成が保たれているか」という視点は、そのまま実務レビューの中心基準になる。

法則、境界、不変条件。

これらは数学の飾りではなく、レビューの判断軸である。

### まとめ

AI 時代のレビューは、読む量を増やすことではない。

見る場所を変えることである。

コードの自然さではなく、構造の持続可能性を見る。

それが人間に残る、もっとも重要な仕事になる。
