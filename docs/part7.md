# Part 9：抽象の起源

ここまでで本書の主線はひととおり通り終えた。

実務と AI 協働まで見たうえで、なお残る読者のために、
最後にもう一段だけ抽象側へ戻る。

ここまで本書では、Functor、Monad、Effect と進んできた。

すると自然に、次の疑問が生まれる。

> これらの抽象は、誰かが美しさのために発明したのか

答えは、ほとんど逆である。

抽象は、現実の設計圧から押し出される。

合成したい。
文脈を壊したくない。
実行を遅らせたい。
解釈を後回しにしたい。

そうした要求を追い詰めると、後から名前がつく。

Part 9 の目的は、定義を覚えることではない。

抽象がどのような必然から現れるのかを掴むことにある。

なお、この Part は本書の中でも抽象度が高い。

Part 8 までで十分に手応えを得られているなら、
ここは「必要になったときに戻る補論」として読んでもよい。

重要なのは用語を覚えることではなく、
設計上の圧力が抽象を呼び出すという感覚を持つことである。

## Chapter 40：随伴の直感

### 問題提起

抽象を学び始めると、多くの人が「左随伴」「右随伴」という言葉で止まる。

それは無理もない。

定義だけを見ると、あまりに遠いからだ。

しかし設計上の直感は、そこまで遠くない。

たとえば我々は、しばしば次のような往復をする。

- 生の値を、より構造化された世界へ持ち上げる
- 構造化された世界で計算する
- 必要なら、ある形で見える値へ戻す

ここで解きたい設計上の痛みは明確である。

- 生の値のままでは失敗や文脈をうまく扱えない
- 構造を足すと計算はしやすくなるが、観察の仕方も必要になる
- この往復に一貫した見取り図がないと、抽象がただのラッピングに見える

この「行って帰る」関係の背後にある整合性を、圏論は随伴として捉える。

### 直感的説明

随伴を最初から記号で見る必要はない。

まずは、

> ある世界では扱いにくいものを、別の世界へ移すと自然に扱える

という感覚を持てばよい。

たとえば、素の値の世界では失敗をうまく表現できない。

だから `Option` や `Result` の世界へ持ち上げる。

そこでは `map` や `flatMap` が自然に現れる。

つまり抽象とは、難しい世界を簡単にするための座標変換でもある。

### 抽象の導入

随伴の直感は、「自由に構成する側」と「忘れて観察する側」の対にある。

一方は構造を加える。

もう一方は、その構造を忘れて中身を見る。

この組がうまく噛み合うとき、構造の導入は恣意的なラッピングではなくなる。

それは、最小限の追加で必要な計算能力を得る操作になる。

ここで重要なのは、随伴を実務テクニックとして暗記することではない。

むしろ、

- ある側では自由に構造を作る
- もう一方ではその構造を忘れて元の世界として読む

という対応があるとき、抽象は突然の思いつきではなく、
問題から自然に現れるものとして見えてくる、という点である。

### 数学的補足

随伴は通常、圏 `C` と `D` の間の関手 `F: C -> D` と `G: D -> C` に対して、

```text
Hom_D(F(A), B) ≅ Hom_C(A, G(B))
```

という自然同型で定義される。

本書でここを詳しく展開することはしない。

重要なのは、Monad のような抽象が突然現れるのではなく、
構造を加える側と、それを忘れて元の世界へ戻して読む側の対応の中から
見えてくる、という見取り図である。

### まとめ

随伴は高級な記号ではない。

抽象が恣意的ではなく、構造を加える側と忘れて読む側の対応から
自然に立ち上がることを示すための補助線である。

## Chapter 41：Free構造

### 問題提起

現実の設計では、すぐに実行したくない計算がある。

- まだ評価戦略を決めたくない
- まだ依存先を確定したくない
- まずは命令列そのものを組み立てたい

つまりここでの痛みは、
「設計を先に確定したいのに、普通に書くと実装が早すぎる」
ことである。

しかし普通に関数を書き始めると、その瞬間に実装が混じる。

ここで必要になるのは、「意味づけ前の構造」だ。

### 直感的説明

Free 構造の直感は単純である。

> 必要最小限の規則だけで、構文を先に作る

ということだ。

たとえば「ログする」「ユーザーを読む」「結果を返す」といった命令があるとする。

このとき、すぐに `console.log` や `fetch` を呼ぶのではなく、
まずは「そういう命令が並んだプログラム」を値として作る。

すると後から、

- テスト用に解釈する
- 本番用に解釈する
- 可視化用に解釈する

ことができる。

### 抽象の導入

Free とは、ある構造を最小限の条件だけで自由に生成したものだと理解できる。

設計上の意味で言えば、

> 実行意味をまだ固定せず、合成規則だけを先に持つ

ということになる。

ここで大切なのは、抽象を増やすことではない。

解釈の責務を後ろへ送ることで、設計と実装を分離することだ。

### TypeScriptコード

```ts
type Program<A> =
  | { type: "pure"; value: A }
  | { type: "log"; message: string; next: Program<A> }
  | { type: "readUser"; id: string; next: (name: string) => Program<A> }

const pure = <A>(value: A): Program<A> => ({
  type: "pure",
  value,
})

const log = <A>(message: string, next: Program<A>): Program<A> => ({
  type: "log",
  message,
  next,
})
```

これは完全な Free Monad 実装ではない。

だが、「命令の意味」ではなく「命令の構文」を先に持つ、という直感はよく出ている。

### 数学的補足

Free 構造は、与えられたデータから最小限の追加条件である種の代数的構造を生成する。

重要なのは「自由」という語の意味である。

何でもできるという意味ではない。

余計な関係式をまだ入れていない、という意味である。

### まとめ

Free 構造は抽象の遊びではない。

解釈より先に合成を確立したいときに現れる、きわめて実務的な考え方である。

## Chapter 42：DSL設計

### 問題提起

抽象を学ぶ最終的な意味は、既存ライブラリの理解ではない。

自分で問題領域の言語を設計できるようになることである。

ここまで来ると、読者が本当に欲しいのは抽象名ではなく、
「自分の現場の問題を、どの語彙で切り分けるか」の感覚である。

多くのコードベースが壊れるのは、ドメインの言葉がないからだ。

その結果、

- 生の `string`
- 生の `boolean`
- 雑な options object
- 暗黙の呼び出し順

で業務ルールを運ぶことになる。

これでは構造が見えない。

### 直感的説明

DSL とは、小さな専用言語である。

ここでいう言語とは、パーサや独自構文を持つ大げさなものとは限らない。

むしろ実務では、

> ドメインで許された操作だけを表現できる API

として現れることが多い。

良い DSL は、利用者に自由を与えすぎない。

書けることを増やすのではなく、書いてはいけないことを減らす。

### 抽象の導入

DSL 設計の核心は、構文と解釈の分離にある。

まず、ドメイン上で意味のある操作を記号化する。

次に、その記号列をどう実行するかを別で与える。

この分離があると、

- テストしやすい
- 変更しやすい
- 意図が読みやすい

という三つが同時に起こる。

ここで Free 構造が役に立つ。

なぜなら DSL はしばしば、「命令の構文木を作って、あとで解釈する」形を取るからだ。

### TypeScriptコード

```ts
type UserCommand =
  | { type: "createUser"; name: string; email: string }
  | { type: "sendWelcomeMail"; userId: string }

type Workflow =
  | { type: "step"; command: UserCommand; next?: Workflow }

const createUser = (
  name: string,
  email: string
): Workflow => ({
  type: "step",
  command: { type: "createUser", name, email },
})
```

これも最小例だが、考え方ははっきりしている。

`fetch` や SQL を直接書く前に、まず「この業務では何が操作として存在するのか」を定義している。

### 数学的補足

DSL は、ドメインの対象と射をコード上で明示化する試みと読める。

この意味で DSL 設計は、単なる API 設計ではなく、小さな圏の設計に近い。

### まとめ

抽象の終着点は、難しい言葉を知ることではない。

問題領域に対して、その領域固有の構文と合成規則を与えられるようになることだ。

それができれば、コードは実装の寄せ集めではなく、意味を持つ言語になる。
