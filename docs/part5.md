# Part 5：型と論理

Part 4 までで、我々は合成を壊さないための器を手に入れた。

しかし、器だけでは設計にならない。

何を `Option` にし、何を `Result` にし、どこで失敗を許さず、どこで状態を露出させるのか。

その判断は、型を「データの入れ物」としてではなく、
制約と分岐を担う境界として見ない限り定まらない。

この Part では、まず型を制約として読み直す。

そのうえで ADT と不変条件まで進み、
なぜ型が論理として働くのかを見ていく。

> 型は、実装を飾る注釈ではない
> 型は、許される接続を絞り込む境界である

---

## Chapter 32：型は制約である

### 問題提起

初学者は型を「値の種類」だと学ぶ。

もちろんそれは間違いではない。

だが、その理解だけでは設計が弱い。

たとえば次のコードを見る。

```ts
type User = {
  name: string
  email: string
}

const sendWelcomeMail = (user: User): string =>
  `send to ${user.email}`
```

このとき設計者が本当に言いたいのは、`name` や `email` が「文字列である」ことだけではない。

むしろ、

- `email` は空文字であってほしくない
- `name` は表示可能な名前であってほしい
- `User` は送信可能な状態であってほしい

という制約である。

型が弱いと、制約はコメントへ落ちる。

コメントへ落ちた制約は、合成の接続点から消える。

### 直感的説明

型とは、何でも入る箱ではない。

何が入ってよいかを限定する境界である。

このとき重要なのは、「表現できる」ことより「表現できない」ことだ。

`string` は広すぎる。

メールアドレスも空文字も `"abc"` も全部入る。

だから `string` にしてしまうと、意味上は別物である値どうしが、同じ接続点に流れ込む。

良い型は、自由を増やさない。

間違った合成の可能性を減らす。

### 抽象の導入

ここで型を、値分類ではなく制約として読む。

```text
型 = この関数が受け取ってよい前提
```

ということである。

この読み方に立つと、型設計は「便利そうな共通型を作る」作業ではなくなる。

むしろ、

1. どの値を区別すべきか
2. どの失敗を接続点から排除すべきか
3. どの前提を型で要求すべきか

を決める作業になる。

### TypeScriptコード

```ts
type Email = string & { readonly _tag: "Email" }
type DisplayName = string & { readonly _tag: "DisplayName" }

type User = {
  name: DisplayName
  email: Email
}

const sendWelcomeMail = (user: User): string =>
  `send to ${user.email}`
```

このコードは実行時検証を省略している。

しかし設計意図は明確である。

`string` をそのまま流すのではなく、「何として検査済みか」を型に残している。

### 数学的補足

型を命題、値をその証明とみなす視点は、Curry-Howard 対応として知られている。

ただし TypeScript は定理証明系ではない。

そのため、この段階では型をそのまま論理体系と同一視する必要はない。

まず重要なのは、「型は前提を表し、型チェックはその局所的検証である」という読み方である。

論理としての手応えは、次の ADT と不変条件でよりはっきりしてくる。

### まとめ

型はデータの種類ではある。

しかし設計上は、それ以上に制約である。

よい型とは、受け取ってはいけないものを入れられない型である。

## Chapter 33：ADT

### 問題提起

現実のドメインは、たいてい分岐を含む。

注文は「下書き」かもしれないし「確定済み」かもしれない。

支払いは「未払い」かもしれないし「支払済み」かもしれない。

にもかかわらず、次のように全部を一つの object で受ける設計は多い。

```ts
type Order = {
  status: string
  paidAt?: Date
  shippedAt?: Date
}
```

この設計の問題は、柔軟すぎることだ。

`status = "draft"` なのに `shippedAt` が入っていても止められない。

ここで必要なのは「if を頑張る」ことではない。

不可能な状態を型で分離することだ。

### 直感的説明

ADT は Algebraic Data Type、つまり和と積で作る型である。

積は「同時に持つ」を表す。

和は「どちらかである」を表す。

業務上の状態分岐は、ほとんどが和である。

だから一枚の大きな object に押し込めるより、状態ごとに分けたほうが自然になる。

### 抽象の導入

和型を導入すると、分岐は値レベルの偶然ではなく、型レベルの構造になる。

すると呼び出し側は、

- どの状態を受け取っているか
- どの分岐を処理しなければならないか

を型から読める。

これが ADT の効能である。

### TypeScriptコード

```ts
type DraftOrder = {
  type: "draft"
  id: string
}

type PaidOrder = {
  type: "paid"
  id: string
  paidAt: Date
}

type ShippedOrder = {
  type: "shipped"
  id: string
  paidAt: Date
  shippedAt: Date
}

type Order =
  | DraftOrder
  | PaidOrder
  | ShippedOrder
```

ここでは `status: string` よりも、状態がはるかに狭く定義されている。

状態遷移の誤配線を、早い段階で止めやすくなる。

### 数学的補足

積型は直積、和型は余積として読める。

プログラミングにおける ADT は、「ドメインの分岐構造を、そのまま型の代数へ写す」試みである。

抽象というより、現実の構造の復元である。

### まとめ

ADT は難しい技巧ではない。

分岐を `string` と `if` に埋め込まず、型の形として露出させるための基本手段である。

## Chapter 34：不変条件

### 問題提起

型を導入しても、まだ壊れることがある。

たとえば `NonEmptyArray` を名乗る値が、本当に空でないとは限らない。

`Email` 型を名乗る値も、検証なしに `as` で作ればただの文字列である。

つまり問題はこうである。

> どこでその型を作ることを許すのか

型名を増やすだけでは、不変条件は守られない。

生成規則まで設計しなければならない。

### 直感的説明

不変条件とは、その値が常に満たしていなければならない条件である。

たとえば、

- 空でない配列
- 0以上の金額
- 支払済みなら `paidAt` を持つ注文

などである。

大切なのは、不変条件は利用側で毎回確かめるものではないということだ。

値を作る瞬間に確立し、その後は前提として使えるようにするべきである。

### 抽象の導入

ここで設計が二層に分かれる。

1. 生の入力を受け取る層
2. 検証済みの値だけを扱う層

この分離がないと、すべての関数が疑心暗鬼になる。

逆にこの分離があれば、検証済みの型を受け取る関数は合成しやすくなる。

### TypeScriptコード

```ts
type NonEmptyArray<A> = [A, ...A[]]

const head = <A>(xs: NonEmptyArray<A>): A =>
  xs[0]

type Positive = number & { readonly _tag: "Positive" }

const addTax = (amount: Positive): number =>
  amount * 1.1
```

ここではコンストラクタ関数を省略しているが、意図ははっきりしている。

「検査前の値」と「検査後の値」を同じ型にしないことである。

### 数学的補足

不変条件は、部分集合や述語つき集合として理解できる。

設計上の要点は、制約された部分を一つの新しい対象として扱い、その対象の上で合成することにある。

### まとめ

不変条件は、関数の中に散らすものではない。

値の生成境界に押し込めるものである。

そうしてはじめて、後続の合成は軽くなる。

## Chapter 35：ドメインモデリング

### 問題提起

ここまでで見てきた

- 型は制約である
- ADT で分岐を露出する
- 不変条件は生成境界で守る

という話は、結局どこへ向かうのか。

その答えがドメインモデリングである。

### 直感的説明

ドメインモデリングとは、現実の名詞をそのままコードへ写すことではない。

業務上区別すべきものを区別し、
許される状態と遷移を明確にすることだ。

たとえば「注文」をただの object として持つのではなく、

- 下書き
- 支払済み
- 発送済み

を別の状態として表せば、
不正な操作は最初から書きにくくなる。

### 抽象の導入

ここで Part 5 の議論が一つにつながる。

- ADT は状態や分岐を露出する
- 不変条件は値の生成境界を固定する
- 型は、許される操作の範囲を狭める

ドメインモデリングとは、
これらを使って問題領域の構造をコード上に回復することだ。

### TypeScriptコード

```ts
type DraftOrder = {
  type: "draft"
  id: string
}

type PaidOrder = {
  type: "paid"
  id: string
  paidAt: Date
}

type Order =
  | DraftOrder
  | PaidOrder

const markAsPaid = (
  order: DraftOrder,
  paidAt: Date
): PaidOrder => ({
  type: "paid",
  id: order.id,
  paidAt,
})
```

ここでは「どの値にも支払い処理できる」のではなく、
`DraftOrder` にしか支払いを適用できない。

つまり業務ルールが、コメントではなく型に埋め込まれている。

### 数学的補足

ドメインモデリングは、問題領域の対象と、その間で許される変換を明示する作業と読める。

重要なのは用語ではない。

壊れやすい自然言語仕様を、合成可能な構造へ落とし直すことにある。

### まとめ

ドメインモデリングは別の話題ではない。

型、ADT、不変条件を、実際の問題領域へ戻した総称である。

そして次の Part では、この構造を副作用を含む計算へどう接続するかを見る。
