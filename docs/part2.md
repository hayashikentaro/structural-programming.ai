# Part 2：壊れる理由

Part 1 で我々は、プログラムを写像の合成として見た。

この視点を採用すると、多くの現象の名前が変わる。

たとえばバグは、「不注意」や「想定漏れ」ではなくなる。

より正確には、

> 合成可能でないものを、合成可能であるかのように扱った結果

として理解される。

この Part では、その壊れ方を分類する。

分類といっても、現象論を並べるのではない。

部分関数、例外、状態。

一見ばらばらに見える三者が、実は同じ構造的問題の別名であることを示す。

つまり、

> 型に現れていないものが、合成を壊す

ということである。

ここを理解しない限り、Functor も Monad も単なる記法に見えてしまう。

だが実際には、それらは壊れた合成を回復するための形式である。

---

## Chapter 15：なぜプログラムは壊れるのか

### 問題提起

壊れるコードにはいくつもの見た目がある。

- `undefined` が飛び出す
- 例外が途中で落ちる
- 同じ関数なのに結果が変わる

通常これらは別々の問題として扱われる。

しかし、その理解では浅い。

現場では名前が違っても、壊れ方の芯は同じだからだ。

たとえば次の3つを見てほしい。

```ts
const head = (xs: number[]): number =>
  xs[0]
```

```ts
const parseNumber = (s: string): number => {
  const n = Number(s)
  if (Number.isNaN(n)) {
    throw new Error("invalid number")
  }
  return n
}
```

```ts
let count = 0

const nextCount = (): number => {
  count += 1
  return count
}
```

これらは違う問題に見える。

だが構造として見れば、すべて同じ問いに行き着く。

> その関数は、本当に型が主張している写像なのか

### 直感的説明

`head` は空配列で壊れる。

`parseNumber` は不正文字列で例外を投げる。

`nextCount` は引数がないのに、呼ぶたびに結果が変わる。

何が起きているのか。

それは、関数の外側にある条件が、型に出ていないということである。

- `head` には「空でないこと」という前提がある
- `parseNumber` には「文字列が数値として読めること」という前提がある
- `nextCount` には「外部状態がこうであること」という隠れた入力がある

にもかかわらず、型はそれを語っていない。

その結果、合成の接続点で必要な情報が失われる。

プログラムが壊れるのは、ロジックが複雑だからではない。

接続条件が偽装されるからである。

### 抽象の導入

ここで、壊れ方を一段抽象化する。

関数が壊れるのではない。

壊れているのは、`A -> B` という宣言である。

つまり本当は、

- `A -> B` ではなく `A -> Option<B>` かもしれない
- `A -> B` ではなく `A -> Result<E, B>` かもしれない
- `A -> B` ではなく `S -> [S, B]` かもしれない

にもかかわらず、それを `A -> B` のように見せている。

もちろん、状態を持つ計算の一般形は通常 `A × S -> B × S` と読む方が正確である。

ここでは「状態が隠れた入力と出力になっている」という点だけを強調するため、
入力値 `A` を省いた簡約形として `S -> [S, B]` を書いている。

すると、次の関数は偽の出力を前提に接続される。

これが合成の失敗である。

したがって設計上の仕事は、壊れた後に `if` を足すことではない。

本来の写像の形を回復することだ。

### TypeScriptコード

```ts
const head = (xs: number[]): number =>
  xs[0]

const toLabel = (n: number): string =>
  `value = ${n}`

const firstLabel = toLabel(head([]))
```

このコードの見た目は単純だが、構造は破綻している。

```text
head: number[] -> number
toLabel: number -> string
```

と読めるから合成してしまうが、実際の `head` は空配列で `number` を返さない。

つまり問題は `toLabel` ではなく、`head` の型宣言にある。

### 数学的補足

数学で `f: A -> B` と言うとき、通常それは全域関数を意味する。

すべての `a ∈ A` に対して `f(a) ∈ B` が定まっていなければならない。

一部の入力で値が定まらないなら、それは部分関数である。

例外や隠れた状態も、プログラム上ではこの全域性を破る別形式と見なせる。

したがって、型に現れていない破綻条件を露出させることは、数学的厳密さではなく、むしろその回復である。

### まとめ

プログラムが壊れる理由は多様に見える。

しかし構造的には一つである。

型が主張していない条件が、実際の合成を支配している。

この問題設定が、以後の章全体を貫く軸になる。

## Chapter 16：部分関数

### 問題提起

最も素朴で、最も頻出する壊れ方は部分関数である。

たとえば次のコードは、日常的すぎるほど日常的だ。

```ts
const head = (xs: number[]): number =>
  xs[0]
```

配列の先頭を取る。

何の変哲もない。

だが空配列を渡した瞬間、この関数は `number` を返せない。

つまりこの関数は、表向きには全関数の顔をしているが、実際には入力領域の一部で未定義である。

ここで問うべきは、「空配列に対応していなかった」ではない。

> なぜ未定義性が型から消えていたのか

である。

### 直感的説明

部分関数とは、「入力は受け取るが、結果はいつも返せるわけではない関数」である。

`head` はまさにそうだ。

空でない配列なら値を返す。

空配列なら返せない。

このとき本当の境界は、

```text
number[] -> number
```

ではなく、

```text
number[] -> number | undefined
```

に近い。

あるいは、より構造的には

```text
number[] -> Option<number>
```

と書くべきである。

これで初めて、利用側は「先頭がないかもしれない」という事実を接続条件として扱える。

### 抽象の導入

部分関数への対処は、本質的には単純である。

部分性を値として持ち上げ、全関数へ変換する。

つまり、

```text
A ⇀ B
```

のような部分関数を、

```text
A -> Option<B>
```

という全関数に変換する。

これにより、「返せない」という事態は未定義ではなく、`None` という値になる。

重要なのは、問題を消すことではない。

未定義性を構造として扱えるようにすることだ。

### TypeScriptコード

```ts
type Option<A> = A | null

const head = (xs: number[]): Option<number> =>
  xs.length === 0 ? null : xs[0]

const toLabel = (n: number): string =>
  `value = ${n}`

const mapOption = <A, B>(
  f: (a: A) => B,
  oa: Option<A>
): Option<B> =>
  oa === null ? null : f(oa)

const firstLabel: Option<string> =
  mapOption(toLabel, head([10, 20, 30]))
```

この変換により、`head` は嘘をつかなくなる。

そして `toLabel` との合成も、`Option` の文脈を通じて正しい形で回復される。

### 数学的補足

部分関数を全関数として表現する標準的な方法の一つが、余積による拡張である。

直感的には、

```text
Option(B) = B + 1
```

と見なす。

ここで `1` は「値を一つだけ持つ型」であり、失敗の印を表す。

このとき部分関数は、「成功か、失敗か」を返す全関数になる。

圏論的には、部分性を値の側へ押し出して合成可能な世界へ戻している、と読める。

### まとめ

部分関数の問題は、エッジケースの不足ではない。

未定義性を隠したことにある。

`Option` はその隠蔽をやめ、欠落を接続可能な形へ持ち上げる。

## Chapter 17：例外

### 問題提起

部分関数よりもさらに悪質なのが、例外である。

部分関数はまだ `undefined` や `null` として表面に漏れることが多い。

しかし例外は、制御フローそのものを別経路へ逃がす。

たとえば次のような関数を考える。

```ts
const parseNumber = (s: string): number => {
  const n = Number(s)
  if (Number.isNaN(n)) {
    throw new Error("invalid number")
  }
  return n
}
```

この関数の型は `string -> number` に見える。

だが実際には、ある入力で `number` を返さない。

しかも返さないだけではない。

呼び出し元の現在地を飛び越え、非局所的に処理を中断する。

これは単に「失敗するかもしれない」以上の問題である。

合成の形そのものが、型の外に逃げている。

### 直感的説明

例外の厄介さは、失敗が値ではなく制御として表現される点にある。

値なら合成できる。

しかし制御は、途中の接続点を飛ばしてしまう。

`parseNumber` の後ろに `double` や `toLabel` をつなげたとしても、
不正入力ではそこに到達しない。

つまり例外は、

```text
string -> number
number -> string
```

という接続図の外で問題を起こしている。

だから追跡しづらく、だから設計を壊しやすい。

### 抽象の導入

失敗を制御ではなく値として戻す必要がある。

そのための典型が `Result<E, A>` である。

これは、

- 成功なら `A`
- 失敗なら `E`

を返す型である。

形としては、

```text
A -> Result<E, B>
```

であり、例外の非局所性を局所的な値へ引き戻す。

こうして初めて、後続の合成は失敗可能性を含めて正しく記述できる。

### TypeScriptコード

```ts
type Result<E, A> =
  | { type: "ok"; value: A }
  | { type: "error"; error: E }

const parseNumber = (s: string): Result<string, number> => {
  const n = Number(s)
  return Number.isNaN(n)
    ? { type: "error", error: "invalid number" }
    : { type: "ok", value: n }
}

const mapResult = <E, A, B>(
  f: (a: A) => B,
  ra: Result<E, A>
): Result<E, B> =>
  ra.type === "error"
    ? ra
    : { type: "ok", value: f(ra.value) }

const double = (n: number): number =>
  n * 2

const labeled: Result<string, number> =
  mapResult(double, parseNumber("21"))
```

ここで失敗は、もはやジャンプしない。

値として残り、型として残り、合成可能な形に留まる。

### 数学的補足

`Result<E, A>` は、固定した `E` に対して `A + E` 型の一種と見なせる。

成功と失敗を余積として表し、両者を同じレベルの値として扱うのである。

これにより例外は「特別な制御」ではなく「通常の出力の一部」へ降格する。

設計上これは非常に大きい。

なぜなら、合成の法則を値の世界へ取り戻せるからである。

### まとめ

例外の本質は、失敗そのものより、失敗が型の外へ逃げていることにある。

`Result` はその逃走を止める。

失敗を値へ戻すことで、プログラムを再び合成可能な構造へ引き戻す。

## Chapter 18：状態

### 問題提起

部分関数や例外は「返せない」という形で壊れる。

状態は少し違う。

返すことは返す。

だが同じ入力に対して同じ結果を返すとは限らない。

たとえば次の関数を見てほしい。

```ts
let seed = 0

const nextId = (): number => {
  seed += 1
  return seed
}
```

型だけ見れば `() -> number` である。

しかしこれは明らかに嘘だ。

なぜなら結果は、見えていない `seed` の値に依存しているからである。

つまりこの関数は、引数を持たないのではない。

引数を隠しているだけである。

### 直感的説明

状態が問題なのは、「変更するから」ではない。

より本質的には、入出力の境界が偽装されるからである。

`nextId` は本当は

```text
state -> number
```

に近いものだ。

しかも状態を読むだけではなく、次の状態も作っている。

すると実際の形は、

```text
state -> (state, number)
```

と読むべきになる。

この見方を採用すると、状態は魔法ではなくなる。

ただの明示的なデータ受け渡しになる。

### 抽象の導入

状態を扱うときの原則は一つである。

> 隠すな。渡せ。

状態をグローバル変数やミュータブルな閉包に閉じ込めると、関数の本当の境界が消える。

それを明示的に引数と返り値へ出せば、状態遷移は合成可能になる。

この形を抽象化したものが `State<S, A>` である。

```text
State<S, A> = S -> (S, A)
```

ここで `S` は状態、`A` は計算結果である。

重要なのは、「状態をなくす」ことではない。

状態変化を写像として表現し直すことだ。

### TypeScriptコード

```ts
type State<S, A> = (state: S) => [S, A]

const nextId: State<number, number> = (state) => {
  const next = state + 1
  return [next, next]
}

const run = nextId(0)

const nextState = run[0]
const generatedId = run[1]
```

このコードでは、もはや隠れた入力はない。

状態遷移が値として表れ、結果とともに返る。

そのため、どの状態から始まり、どの状態へ移るかを、合成の接続点として扱える。

### 数学的補足

状態を明示化した `S -> (S, A)` という形は、単なる実装テクニックではない。

これは「計算とは、状態を受け取って新しい状態と結果を返す写像である」という再定式化である。

関数型プログラミングにおいて状態が扱いやすくなるのは、不変だからではなく、その遷移形が型に露出しているからである。

つまり重要なのはミュータビリティの有無ではなく、境界の可視性である。

### まとめ

状態の問題は、副作用という言葉だけでは足りない。

本質は、隠れた入力と隠れた出力である。

`State` はそれを可視化し、状態変化を合成可能な写像へ戻す。

## Chapter 19：統一的理解

### 問題提起

ここまで、

- 部分関数
- 例外
- 状態

を別々に見てきた。

しかし、もしここで「状況に応じて使い分けるテクニック集」としか見えていないなら、まだ浅い。

必要なのは統一的理解である。

つまり、これらが何を共有しているかを掴むことだ。

### 直感的説明

3つを並べると、違いより共通点の方が大きい。

| 現象 | 隠れているもの |
|------|----------------|
| 部分関数 | 値がないかもしれないという条件 |
| 例外 | 失敗するかもしれないという分岐 |
| 状態 | 入力と出力に関わる外部文脈 |

どれも、「コードの外側に追いやられた条件」が原因である。

つまり壊れているのは個々の if 文ではない。

関数境界の表現力である。

ここで視点を統一すると、対処も統一される。

- 欠落は `Option`
- 失敗は `Result`
- 状態は `State`

として型へ戻す。

すると、ばらばらだった問題群が「文脈つきの値」として同じ平面へ並ぶ。

### 抽象の導入

ここで重要な観察がある。

`Option<A>`、`Result<E, A>`、`State<S, A>` は形こそ違うが、どれも「ただの `A` ではない」。

そこには何らかの文脈がついている。

- 欠落の文脈
- 失敗の文脈
- 状態遷移の文脈

そして、我々が後で学ぶ `map` や `flatMap` は、この文脈を壊さずに合成するために現れる。

つまり抽象は後付けの飾りではない。

壊れ方の共通原因を見抜いたときに、必然的に現れる形式である。

### TypeScriptコード

```ts
type Option<A> = A | null

type Result<E, A> =
  | { type: "ok"; value: A }
  | { type: "error"; error: E }

type State<S, A> = (state: S) => [S, A]

const safeHead = (xs: number[]): Option<number> =>
  xs.length === 0 ? null : xs[0]

const parseNumber = (s: string): Result<string, number> => {
  const n = Number(s)
  return Number.isNaN(n)
    ? { type: "error", error: "invalid number" }
    : { type: "ok", value: n }
}

const nextId: State<number, number> = (state) => {
  const next = state + 1
  return [next, next]
}
```

この3つは用途が違う。

しかし設計上はどれも、

> 素の値 `A` を、そのまま返してはいない

という点で一致している。

ここが見えれば、抽象化の方向は自然に定まる。

### 数学的補足

圏論の言葉を先取りして言えば、これらはそれぞれ異なる型構成子であり、値をある文脈へ包み込む。

重要なのは、包むことそのものではない。

包まれた値に対して、どのような射の持ち上げと合成が可能かである。

したがって本質は「箱」ではなく「構造保存」である。

### まとめ

部分関数、例外、状態は別問題ではない。

型に現れていない文脈が、合成を壊すという一つの問題である。

この統一的理解が、Effect や Monad を受け入れる地盤になる。

## Chapter 20：Effect

### 問題提起

ここまでで、壊れた合成を守るために

- `Option`
- `Result`
- `State`

を導入してきた。

では、これらを総称する視点はあるのか。

もしないなら、毎回新しい型が出てくるたびに、一から理解し直さなければならない。

それでは抽象になっていない。

必要なのは、

> それはどんな Effect なのか

と問える視点である。

### 直感的説明

Effect とは、ここでは単に「副作用っぽい何か」という曖昧語ではない。

ここでの Effect は広い意味での語であり、
外部世界との入出力だけを指しているわけではない。

より正確には、

> 素の値だけでは表現しきれない文脈を伴った計算の形

である。

`Option<A>` には欠落の文脈がある。

`Result<E, A>` には失敗の文脈がある。

`State<S, A>` には状態遷移の文脈がある。

これらはすべて、「ただ `A` を返す計算」ではない。

だが同時に、「もう合成できない混沌」でもない。

適切な演算を与えれば、文脈つきのまま合成できる。

その意味で Effect とは、合成を壊すものではなく、合成の条件を増やすものである。

後の Part 6 では、この広い意味での Effect のうち、
とくに外部世界と関わる effect を取り出して扱う。

### 抽象の導入

ここで視点を一段上げる。

普通の関数は

```text
A -> B
```

だった。

しかし Effect を持つ計算は、たとえば

```text
A -> F<B>
```

の形を持つ。

ここで `F` は文脈である。

この `F` が何であるかによって、欠落、失敗、状態、非同期などが区別される。

重要なのは、この形に入った瞬間、もはや通常の関数合成だけでは足りないことだ。

だから `map` が必要になり、後には `flatMap` が必要になる。

Effect は複雑さではない。

複雑さを、合成可能な構造として記述し直した結果である。

### TypeScriptコード

```ts
type Option<A> = A | null

type Result<E, A> =
  | { type: "ok"; value: A }
  | { type: "error"; error: E }

type State<S, A> = (state: S) => [S, A]

const findUserName = (id: number): Option<string> =>
  id === 0 ? null : "Ada"

const parseAge = (s: string): Result<string, number> => {
  const n = Number(s)
  return Number.isNaN(n)
    ? { type: "error", error: "invalid age" }
    : { type: "ok", value: n }
}

const allocateId: State<number, number> = (state) => {
  const next = state + 1
  return [next, next]
}
```

これらは一見まったく違う。

しかし「値を返す」のではなく、「文脈つきの値を返す」という点では同じである。

この同型的な見方ができると、個別技法の背後にある共通構造が見え始める。

### 数学的補足

Effect という語は文脈依存だが、本書では「計算をそのままの値ではなく構造化された文脈の中で表すもの」として使う。

圏論的には、自己関手やモナドの言葉で整理されることが多い。

ただしこの段階では、定義を急ぐ必要はない。

重要なのは、Effect が「不純だから危険」なのではなく、「文脈つきだから通常の合成規則を拡張する必要がある」と理解することだ。

### まとめ

Effect とは、値に付着した余計な装飾ではない。

計算が本来持っていた文脈を、隠さず型へ現した結果である。

この理解があれば、次に学ぶ `flatMap` は難しい操作ではなくなる。

それは単に、

> `A -> F<B>` どうしをどう合成するか

という必然的な問いへの答えだからである。

## Chapter 21：まとめ

Part 2 で扱ったのは、プログラムが壊れる具体例ではない。

壊れる理由の形式である。

まず見たのは、壊れ方の背後には共通した構造があるということだった。

部分関数は、未定義性を隠していた。

例外は、失敗を制御フローへ逃がしていた。

状態は、入出力境界の外に依存関係を隠していた。

これらはいずれも、

> 型に現れていないものが、合成を壊す

という一点で一致している。

そのため対処もまた統一される。

欠落は `Option` に、失敗は `Result` に、状態遷移は `State` に持ち上げる。

つまり問題を消すのではなく、文脈として型へ露出させる。

ここに本書の一貫した立場がある。

良い設計とは、分岐を消すことでも、例外を減らすことでも、可変状態を道徳的に否定することでもない。

> 合成条件を隠さないこと

である。

AI 時代の実務でも、この視点は決定的に重要である。

AI は局所的な実装を書ける。

だが、

- どの失敗を型へ持ち上げるか
- どの文脈を明示するか
- どの合成規則を採用するか

は構造設計の問題であり、人間の責任である。

次の Part では、Functor を扱う。

それは抽象名を覚える章ではない。

むしろ、

> 文脈を保ったまま写像を適用するにはどうすればよいか

という、ここまでの必然から生まれる章である。
